# Server-Side Lineup Carry-Forward Implementation Plan

**Goal:** Ensure all teams score points every gameweek, even if users don't log in, by carrying forward previous lineups server-side during aggregation.

**Architecture:** Add `ensureLineups()` function to `lib/scoring/pipeline.ts` that runs inside `aggregateGameweek()` before the scoring loop, inside the same `$transaction` for atomicity. Creates real Lineup + LineupSlot records for teams missing a lineup.

**Tech Stack:** Next.js 15, Prisma 6, TypeScript, Vitest

**Scope:** 2 phases from design

**Codebase verified:** 2026-03-27

---

## Acceptance Criteria Coverage

This phase implements:

### server-side-lineup-carry-forward.AC1: Carry-Forward from Previous Gameweek
- **server-side-lineup-carry-forward.AC1.1 Success:** Team with GW1 lineup but no GW2 lineup gets GW1 lineup cloned to GW2 (same XI, bench, captain/VC, bench priority)
- **server-side-lineup-carry-forward.AC1.2 Success:** Carry-forward picks the most recent previous lineup (GW2 over GW1 if both exist)
- **server-side-lineup-carry-forward.AC1.3 Success:** Carried-forward lineup creates real Lineup + LineupSlot records in DB
- **server-side-lineup-carry-forward.AC1.4 Failure:** Team with existing lineup for this GW is not modified
- **server-side-lineup-carry-forward.AC1.5 Success:** Players removed from squad since previous lineup are excluded from carry-forward slots

### server-side-lineup-carry-forward.AC3: Pipeline Integration
- **server-side-lineup-carry-forward.AC3.1 Success:** `ensureLineups()` runs inside `aggregateGameweek()` within the `$transaction`, before the scoring loop
- **server-side-lineup-carry-forward.AC3.2 Success:** After `ensureLineups()`, teams that previously had no lineup now have one and score points through the normal pipeline

---

**Note on GET endpoint discrepancy:** The existing carry-forward in `app/api/teams/[teamId]/lineups/[gameweekId]/route.ts:52-58` only looks at the immediately previous gameweek (`number - 1`). `ensureLineups()` looks at the most recent lineup across ALL previous gameweeks. This is intentional — `ensureLineups` is a safety net that handles multi-week gaps (user absent for 2+ weeks). The GET endpoint behavior is a follow-up improvement outside this PR's scope.

### Task 1: Create `ensureLineups()` with carry-forward logic and pipeline integration

**Verifies:** server-side-lineup-carry-forward.AC1.1, server-side-lineup-carry-forward.AC1.2, server-side-lineup-carry-forward.AC1.3, server-side-lineup-carry-forward.AC1.4, server-side-lineup-carry-forward.AC1.5, server-side-lineup-carry-forward.AC3.1

**Files:**
- Modify: `lib/scoring/pipeline.ts` — add `ensureLineups()` function and call it inside `aggregateGameweek()`

**Implementation:**

Add `ensureLineups()` to `lib/scoring/pipeline.ts` (after the `aggregateGameweek` function, before the helpers section at line 486).

The function accepts a Prisma transaction client (`tx`) so it runs atomically with the scoring loop:

```typescript
type TeamsWithLineups = Awaited<ReturnType<typeof prisma.team.findMany<{
  include: { league: true; lineups: { where: { gameweekId: string }; include: { slots: true } }; chipUsages: { where: { gameweekId: string; status: 'PENDING' } } }
}>>>
```

Function logic:
1. Iterate through all teams
2. Skip teams where `team.lineups[0]` already exists (AC1.4)
3. For teams without a lineup: query `tx.lineup.findFirst()` with `where: { teamId: team.id }`, `orderBy: { gameweek: { number: 'desc' } }`, `include: { slots: true, gameweek: { select: { number: true } } }` to find most recent previous lineup (AC1.2)
4. If previous lineup found:
   a. Query `tx.teamPlayer.findMany({ where: { teamId: team.id }, select: { playerId: true } })` to get current squad
   b. Filter previous lineup slots to only include players still in squad (AC1.5)
   c. Create new Lineup + filtered LineupSlot records via `tx.lineup.create()` (AC1.1, AC1.3)
   d. Log: `console.log(\`ensureLineups: carried forward ${team.name} from GW${prevLineup.gameweek.number}\`)`
   e. Push the new lineup into `team.lineups` array so scoring loop picks it up
   f. Increment `carriedForward` counter
5. Return `{ carriedForward, autoGenerated: 0 }` (auto-generate added in Phase 2)

**Integration into `aggregateGameweek()`:**

Move the `ensureLineups()` call INSIDE the `$transaction` block. In `aggregateGameweek()`, change line 398 onwards:

```typescript
await prisma.$transaction(async (tx) => {
  // Carry-forward lineups for teams missing one this GW
  const ensureResult = await ensureLineups(tx, teams, gameweekId)
  if (ensureResult.carriedForward > 0 || ensureResult.autoGenerated > 0) {
    console.log(`ensureLineups: ${ensureResult.carriedForward} carried forward, ${ensureResult.autoGenerated} auto-generated`)
  }

  for (const team of teams) {
    // ... existing scoring loop unchanged
  }
  // ... rest of transaction unchanged
})
```

The function signature uses the transaction client:
```typescript
async function ensureLineups(
  tx: Parameters<Parameters<typeof prisma.$transaction>[0]>[0],
  teams: TeamsWithLineups,
  gameweekId: string
): Promise<{ carriedForward: number; autoGenerated: number }>
```
